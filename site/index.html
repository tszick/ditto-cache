Distributed cache, without the drama

Fast cache. Clear behavior.

Ditto is a strongly consistent distributed in-memory key-value cache built to stay predictable under real usage. The fast path matters, but so do the daily touchpoints: APIs, watch flows, namespaces, admin UI, CLI, and observability.

Why it matters

Not just a cache engine, but a cache system that feels usable.

Ditto is not built around "it will probably work somehow". It is built around making the cache layer predictable for applications and manageable for operators. The result is something fast, visible, and developer-friendly at the same time.

Predictable reads and writes

Local reads stay fast, while the write model stays strongly consistent. That is useful when you do not want every cache decision to come with a paragraph of caveats.

Comfortable from the app side

Alongside basic GET, SET and DELETE, you get batch writes, watch, TTL updates by pattern, deletes by pattern, and namespace support. Less helper code around the cache, more actual product work.

Clear from the ops side

Node state, health summary, memory usage, evictions, request and error stats, admin UI and CLI together make the system feel manageable instead of mysterious.

Capabilities

The things that make Ditto easy to move with.

TTL and memory limits

Per-key TTL, global default TTL, background sweep and LFU eviction. The cache does not just grow forever, it behaves like a system that expects real load.

Watch flows over TCP

If you want to follow key changes in near real time, watch and unwatch plus pushed watch events are already part of the model.

Pattern-based operations

Delete keys by pattern, or update TTL for a whole key group. Very handy for sessions, feature caches, or tenant-scoped keyspaces.

Namespaces and quota

Namespaces separate the cache logically, while quota helps make sure one tenant or use case cannot crowd everyone else out.

Persistence when you need it

Backup, export and import paths are part of the story too, so Ditto is not limited to purely in-memory thinking. Persistence can be controlled explicitly instead of being an accidental side effect.

HTTP and TCP together

REST API for straightforward integrations, binary TCP when you want a leaner or more interactive model. You do not have to force every client into one style.

Healthy diagnostics

Stats, health summary, latency estimates, error breakdowns, quota pressure, hot-key and repair signals. Not a black box, but an observable component.

Interfaces

You do not need a long explanation to start using it.

On the data side, Ditto gives you two clear entry points: HTTP and TCP. That means teams can use it in the way that feels natural for them.

For operators

The admin side is part of the product, not an afterthought.

`ditto-mgmt` gives you a web UI and admin API, while `dittoctl` gives you a scriptable CLI. So the same cache can stay fast and still remain manageable when multiple nodes, namespaces and use cases are involved.

Dashboard and node control

Real-time cluster view, node states, memory usage, active and syncing states, and the typical controls operators actually need.

Health and metrics mindset

Health summary and stats do not just answer "is it alive", but also show how the cache behaves under pressure and around failure paths.

Security and control

Auth, TLS, policy gates and admin channels push Ditto toward deliberate, controlled usage instead of accidental exposure. The same goes for persistence: backup and restore paths are there, but they stay explicitly governed.

Clients

It is not tied to one language.

The related Ditto clients live in a separate repository, and there is support for Go, Java, Node.js and Python. For this site, the important part is simply that Ditto can fit multi-language teams without drama.
